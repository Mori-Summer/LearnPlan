# 缓存行与伪共享实战指南（C++）

本文目标不是“背概念”，而是建立一条完整链路：

- 硬件如何按缓存行工作
- 为什么这会变成单线程/多线程性能问题
- 常见误区是什么
- 如何用工程手段定位与修复
- 给出可落地的 C++ 范例代码

这篇文档与 `SoA和AoS.md`、`虚拟内存.md` 等文档互补，重点聚焦在“缓存行粒度”对性能和并发行为的影响。

---

## 1. 实际原理：缓存行不是细节，而是性能基本单位

### 1.1 CPU 按“缓存行”搬运，不按“变量”搬运

CPU 从内存读取数据时，最小搬运单位是缓存行（Cache Line），不是 `int`、`double` 或某个 struct 字段。

常见平台缓存行大小：

- x86_64：通常 64B
- ARM64（桌面/服务器）：通常 64B
- 特定架构可能是 128B 或其他大小

结论：

- 你“只读 4 字节”，CPU 也常常会加载整行 64B
- 你“改一个原子变量”，一致性协议也会按整行仲裁所有权

### 1.2 缓存层级与延迟差异

简化视角：

- L1：最小最快，通常每核私有
- L2：较大较慢，通常每核私有或小范围共享
- L3：更大更慢，常见为多核共享
- DRAM：容量大但延迟远高于缓存

性能核心：尽可能让热点数据停留在更近的层级（L1/L2）。

### 1.3 地址映射：为什么会出现冲突失效

缓存不是全相联，通常是组相联（set-associative）。地址会被拆分为：

- `offset`：行内偏移
- `index`：映射到哪一组
- `tag`：区分同组中的不同行

如果多个热点地址反复映射到同一个 set，哪怕缓存总容量足够，也会互相挤掉，形成 conflict miss。

这就是“容量看起来够，但性能仍很差”的常见原因之一。

### 1.4 写策略与 RFO：写入并非免费

主流缓存采用 write-back：

- 写命中：先改缓存，后续再回写内存
- 写缺失：常见策略是先把该行读进来（Read For Ownership, RFO），再写

因此写共享数据时不仅有写操作，还有潜在的“抢行所有权”成本。

### 1.5 多核一致性：MESI/MOESI 的真实成本

以 MESI 简化说明：

- M（Modified）
- E（Exclusive）
- S（Shared）
- I（Invalid）

当核心 A 要写一个 S 状态缓存行时：

1. A 发起 RFO 请求拿写权限
2. 其他核心该行失效为 I
3. A 才能进入 M 并写入

若核心 B 随后也要写同一行，流程再次发生，形成缓存行在核心间来回“乒乓”（cache line ping-pong）。

### 1.6 一致性不等于同步正确性

硬件一致性协议只能保证缓存副本最终一致，不能替代 C++ 语言层并发语义。

- `std::atomic` 保证原子性
- `memory_order` 定义跨线程可见顺序
- 锁/条件变量解决复合不变量与等待关系

“缓存一致性协议能替代同步原语”是典型误解。

## 2. 影响内容：缓存行问题会表现在哪些维度

### 2.1 吞吐、延迟与扩展性

典型症状：

- 线程数从 1 增到 2/4/8，吞吐不升反降
- 平均延迟看似可接受，但 P95/P99 恶化
- 单核快，多核慢

### 2.2 单线程性能：有效载荷比例

如果对象很“胖”，循环只用其中 1-2 个字段，缓存行里大量字节都是无效载荷。

表现：

- 带宽浪费
- cache miss 上升
- SIMD/预取收益下降

### 2.3 多线程性能：真共享与伪共享

- 真共享（True Sharing）：线程确实读写同一变量
- 伪共享（False Sharing）：线程写不同变量，但变量落在同一缓存行

伪共享本质是“物理同线导致的所有权争夺”，不是代码层显式共享。

### 2.4 原子热点与无锁结构

- 全局原子计数器在高并发下常成为一致性热点
- SPSC/MPMC 队列的 `head`/`tail` 若同线，吞吐会被拖垮
- 频繁短事务写共享状态容易触发大量失效广播

### 2.5 系统层副作用

- c2c 传输增加
- 总线/互连流量升高
- 跨 NUMA 节点时成本进一步放大

## 3. 为什么会产生问题：从“代码看起来独立”到“硬件实际冲突”

常见成因可归为 5 类：

1. 地址邻近导致同线
- 两个字段逻辑无关，但物理地址只差几十字节

2. 高频写导致行所有权抖动
- 读共享一般可并存（S）
- 写共享需要独占（M），会触发失效

3. 错误的布局假设
- 仅 `alignas(64)` 对齐起始地址，不保证数组元素彼此不共线

4. 热冷数据混放
- 高频字段与低频大字段混放，浪费缓存容量

5. 访问模式与结构不匹配
- 按字段扫描却使用 AoS，导致行利用率低

## 4. 常见误区（重点反驳）

### 误区 1：用了 `atomic` 就一定快

`atomic` 解决的是原子性与可见性，不保证在高争用下吞吐最好。高并发写同一行时，原子也会被一致性流量拖慢。

### 误区 2：`alignas(64)` 就不会伪共享

`alignas(64)` 只保证对象起始地址对齐。若 `sizeof(T) < 64`，`T arr[N]` 的相邻元素仍可能同线。

### 误区 3：缓存一致性协议能替代锁或内存序

错误。MESI 处理缓存副本一致，不负责你的业务顺序语义。

### 误区 4：只看平均值就能判断优化有效

缓存争用问题常首先体现在尾延迟和扩展曲线上。只看均值容易误判。

### 误区 5：不测量直接改布局

缓存优化高度依赖场景，不测量就重构，容易把可维护性换成无收益复杂度。

## 5. 如何排查：先证明“是缓存行问题”再优化

### 5.1 最小复现与实验纪律

建议流程：

1. 做最小可复现基准（只保留疑似热点）
2. 固定 CPU 频率策略与线程绑核（减少调度噪声）
3. 每轮只改一个变量（布局、对齐、线程数、内存序）
4. 记录 1/2/4/8/... 线程扩展曲线

### 5.2 Linux 常见指标（perf）

```bash
# 先看总体趋势
perf stat -e cycles,instructions,cache-references,cache-misses ./bench

# 看核心间缓存行传输（平台支持时）
perf c2c record -g ./bench
perf c2c report
```

可重点观察：

- miss 比例是否随线程数异常上升
- c2c 传输是否集中在少数地址
- IPC 是否在并发上升时显著下降

### 5.3 结果解释原则

- 优先看趋势，不迷信单次值
- 同时看吞吐和尾延迟
- 对比“单线程最快版本”与“多线程可扩展版本”

## 6. 如何解决：按问题类型选择策略

### 6.1 伪共享修复：隔离写热点

- 使用 `std::hardware_destructive_interference_size`（可用时）
- 对热点写字段做独立对齐与填充
- 把频繁写字段从共享结构拆出

### 6.2 热点分片（Sharding）

把一个全局写热点拆成每线程或每核槽位：

- 写路径：低争用
- 读路径：聚合开销可控

适用：计数器、统计、监控打点。

### 6.3 数据布局重构

- 按字段批处理：优先 SoA
- 按对象多字段处理：AoS 可能更合适
- 热冷分离提高缓存有效载荷比例

### 6.4 降低共享写频率

- 批量提交（batch）代替每次原子更新
- 本地累加 + 周期合并
- 将控制路径写与数据路径读分层

### 6.5 内存序与同步策略

- 仅需计数时可考虑 `relaxed`
- 需要发布-订阅关系时用 `release/acquire`
- 复合状态一致性优先用 mutex，避免“原子拼图”

## 7. 范例代码（错误版与修正版）

说明：以下示例都可独立保存为单文件并编译，示例编译参数建议：

```bash
g++ -std=c++20 -O2 -pthread demo.cpp -o demo
```

### 7.1 示例一：伪共享计数器（错误版 vs 修正版）

错误版（`a` 与 `b` 可能同线）：

```cpp
#include <atomic>
#include <iostream>
#include <thread>

struct Counters {
    std::atomic<long long> a{0};
    std::atomic<long long> b{0};
};

int main() {
    Counters c;
    constexpr int N = 50'000'000;

    std::thread t1([&] {
        for (int i = 0; i < N; ++i) c.a.fetch_add(1, std::memory_order_relaxed);
    });
    std::thread t2([&] {
        for (int i = 0; i < N; ++i) c.b.fetch_add(1, std::memory_order_relaxed);
    });

    t1.join();
    t2.join();
    std::cout << c.a.load() + c.b.load() << "\n";
}
```

修正版（隔离缓存行）：

```cpp
#include <atomic>
#include <cstddef>
#include <iostream>
#include <new>
#include <thread>

#if defined(__cpp_lib_hardware_interference_size)
constexpr std::size_t kLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kLine = 64;
#endif

struct alignas(kLine) PaddedCounter {
    std::atomic<long long> value{0};
};

struct Counters {
    PaddedCounter a;
    PaddedCounter b;
};

int main() {
    Counters c;
    constexpr int N = 50'000'000;

    std::thread t1([&] {
        for (int i = 0; i < N; ++i) c.a.value.fetch_add(1, std::memory_order_relaxed);
    });
    std::thread t2([&] {
        for (int i = 0; i < N; ++i) c.b.value.fetch_add(1, std::memory_order_relaxed);
    });

    t1.join();
    t2.join();
    std::cout << c.a.value.load() + c.b.value.load() << "\n";
}
```

### 7.2 示例二：`alignas` 陷阱（数组元素仍可能共线）

反例：

```cpp
#include <atomic>
#include <cstddef>
#include <iostream>

struct alignas(64) Slot {
    std::atomic<int> x{0};
};

int main() {
    std::cout << "alignof(Slot)=" << alignof(Slot) << "\n";
    std::cout << "sizeof(Slot)=" << sizeof(Slot) << "\n";

    // 如果 sizeof(Slot) < 64，数组元素仍可能共享缓存行
    Slot slots[2];
    std::cout << "distance="
              << (reinterpret_cast<char*>(&slots[1]) - reinterpret_cast<char*>(&slots[0]))
              << " bytes\n";
}
```

更稳妥做法：显式填充，确保 `sizeof(T) >= line size` 且对齐正确。

```cpp
#include <atomic>
#include <cstddef>
#include <new>

#if defined(__cpp_lib_hardware_interference_size)
constexpr std::size_t kLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kLine = 64;
#endif

struct alignas(kLine) Slot {
    std::atomic<int> x{0};
    char pad[kLine - sizeof(std::atomic<int>)];
};

static_assert(sizeof(Slot) >= kLine);
static_assert(alignof(Slot) >= kLine);
```

### 7.3 示例三：全局原子计数器 vs 分片计数器

分片计数器：

```cpp
#include <atomic>
#include <cstddef>
#include <cstdint>
#include <iostream>
#include <new>
#include <thread>
#include <vector>

#if defined(__cpp_lib_hardware_interference_size)
constexpr std::size_t kLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kLine = 64;
#endif

struct alignas(kLine) Shard {
    std::atomic<std::uint64_t> value{0};
};

class ShardedCounter {
public:
    explicit ShardedCounter(std::size_t n) : shards_(n) {}

    void add(std::size_t shard_idx, std::uint64_t delta) {
        shards_[shard_idx].value.fetch_add(delta, std::memory_order_relaxed);
    }

    std::uint64_t sum() const {
        std::uint64_t total = 0;
        for (const auto& s : shards_) {
            total += s.value.load(std::memory_order_relaxed);
        }
        return total;
    }

private:
    std::vector<Shard> shards_;
};

int main() {
    const std::size_t nthreads = std::thread::hardware_concurrency() ?
                                 std::thread::hardware_concurrency() : 4;
    ShardedCounter counter(nthreads);
    constexpr int N = 20'000'000;

    std::vector<std::thread> threads;
    threads.reserve(nthreads);
    for (std::size_t i = 0; i < nthreads; ++i) {
        threads.emplace_back([&, i] {
            for (int k = 0; k < N; ++k) counter.add(i, 1);
        });
    }

    for (auto& t : threads) t.join();
    std::cout << counter.sum() << "\n";
}
```

### 7.4 示例四：SPSC 队列索引同线问题

错误布局：

```cpp
#include <atomic>

struct BadIndices {
    std::atomic<std::size_t> head{0};
    std::atomic<std::size_t> tail{0};
};
```

修正布局：

```cpp
#include <atomic>
#include <cstddef>
#include <new>

#if defined(__cpp_lib_hardware_interference_size)
constexpr std::size_t kLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kLine = 64;
#endif

struct GoodIndices {
    alignas(kLine) std::atomic<std::size_t> head{0};
    alignas(kLine) std::atomic<std::size_t> tail{0};
};

static_assert(alignof(GoodIndices) >= alignof(std::atomic<std::size_t>));
```

在 SPSC 中，生产者主要写 `tail`，消费者主要写 `head`。同线会导致彼此失效；分离后通常可显著改善吞吐。

### 7.5 示例五：AoS 与 SoA 的缓存利用差异

AoS（按字段扫描时有效载荷较低）：

```cpp
#include <cstddef>
#include <vector>

struct Particle {
    float x, y, z;
    float vx, vy, vz;
    int id;
};

double sum_x_aos(const std::vector<Particle>& p) {
    double s = 0;
    for (std::size_t i = 0; i < p.size(); ++i) {
        s += p[i].x;
    }
    return s;
}
```

SoA（按字段扫描更友好）：

```cpp
#include <cstddef>
#include <vector>

struct ParticlesSoA {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
};

double sum_x_soa(const ParticlesSoA& p) {
    double s = 0;
    for (std::size_t i = 0; i < p.x.size(); ++i) {
        s += p.x[i];
    }
    return s;
}
```

如果只做字段批处理，SoA 通常更容易获得更高 cache line 利用率和向量化收益。

### 7.6 示例六：内存序边界（`relaxed` 不是发布协议）

错误示例（用 `relaxed` 做发布，消费者不一定看到配套数据）：

```cpp
#include <atomic>
#include <thread>

std::atomic<int> ready{0};
int payload = 0;

int main() {
    std::thread producer([] {
        payload = 42;
        ready.store(1, std::memory_order_relaxed); // 仅原子，不建立发布关系
    });

    std::thread consumer([] {
        while (ready.load(std::memory_order_relaxed) == 0) {}
        // 这里读取 payload 在语义上不安全
        (void)payload;
    });

    producer.join();
    consumer.join();
}
```

正确示例（release/acquire）：

```cpp
#include <atomic>
#include <thread>

std::atomic<int> ready{0};
int payload = 0;

int main() {
    std::thread producer([] {
        payload = 42;
        ready.store(1, std::memory_order_release);
    });

    std::thread consumer([] {
        while (ready.load(std::memory_order_acquire) == 0) {}
        int observed = payload;
        (void)observed;
    });

    producer.join();
    consumer.join();
}
```

该示例用于说明：缓存一致性并不会自动给你跨变量顺序保证，顺序语义需由语言层内存模型建立。

## 8. 实战检查清单

排查缓存行相关瓶颈时，按下面顺序自检：

1. 是否存在高频写共享字段（尤其是原子）？
2. 这些字段是否可能位于同一缓存行？
3. 能否改为每线程本地写 + 汇总？
4. 热字段与冷字段是否已分离？
5. 访问模式是否匹配布局（AoS/SoA）？
6. 是否做过可复现基准并观察扩展曲线？
7. 是否验证过尾延迟而非仅平均值？

---

## 9. 总结

- 缓存行是 CPU 与内存系统交互的真实粒度，也是并发写争用的真实粒度。
- 伪共享是“逻辑独立、物理同线”导致的所有权争夺。
- MESI 解决硬件一致性，不解决程序语义顺序；正确同步依赖 C++ 并发模型。
- 高质量优化路径是：先测量、再定位、最后用布局与并发结构改造解决问题。
