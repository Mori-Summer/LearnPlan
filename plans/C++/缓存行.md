# 缓存行（Cache Line）与 C++ 多线程性能

这篇文档聚焦一个工程事实：

- CPU 不是按“变量”搬运数据，而是按“缓存行”搬运数据。
- 多线程性能问题里，很多瓶颈不是算法复杂度，而是缓存行争用（尤其是伪共享）。

---

## 1. 缓存行是什么

缓存行是 CPU Cache 和内存之间传输数据的最小粒度。

当 CPU 读取某个地址时，通常会把该地址所在的一整行（line）载入缓存，而不是只读取一个字节或一个变量。

### 1.1 常见大小

- 现代 x86_64：通常 64B
- 大多数 ARM64 服务器/桌面：通常 64B
- 某些平台可能是 128B 或其他值（不要硬编码假设）

实践建议：

- C++17 可优先使用 `std::hardware_destructive_interference_size`（`<new>`）做跨平台保守对齐。
- 若标准库未提供，再回退到经验值（如 64）。

### 1.2 缓存层级（简化）

- L1：最小、最快，通常每核私有
- L2：更大更慢，通常每核私有（有些架构共享）
- L3：更大更慢，常见为多核共享

核心点：缓存命中层级越低（L1 优于 L2 优于 L3 优于内存），延迟越小。

---

## 2. 地址与缓存行的关系

对 line size = 64B 的情况：

- 行基址：`line_base = addr & ~(64 - 1)`
- 行内偏移：`offset = addr & (64 - 1)`

这意味着：

- 地址 `0x1000` 到 `0x103F` 属于同一缓存行。
- 变量并不“天然 64B 对齐”。一个对象可能跨越两个缓存行。

这点很重要：

- “缓存行边界”由地址决定。
- “对象对齐”由类型/分配器决定。
- 二者不一致时，可能出现跨行访问、额外 miss、甚至更差的并发行为。

### 2.1 组相联与冲突失效（Conflict Miss）

缓存通常不是“全相联”，而是组相联（set-associative）：

- 地址会映射到某个 set
- 每个 set 只能容纳有限条 line（way 数）

如果多个热点地址反复映射到同一个 set，即使缓存总容量还够，也会互相挤出，形成冲突失效。

工程意义：

- 连续访问通常友好；
- 某些固定步长访问可能持续冲突；
- 调整数据布局、步长或填充，可能显著降低 miss。

---

## 3. 为什么缓存行会显著影响性能

### 3.1 局部性

- 时间局部性：刚访问过的数据很可能再次访问
- 空间局部性：相邻地址很可能被访问

缓存行正是利用空间局部性，一次带来一小段连续数据。

### 3.2 预取（Prefetch）

硬件预取器会根据访问模式（如顺序遍历）提前加载后续缓存行。

如果访问模式是随机跳跃或大步长 stride，预取效果会变差，缓存 miss 上升。

### 3.3 典型反例

```cpp
// 假设 sizeof(Node) 很大，且我们只需要 value
for (size_t i = 0; i < n; ++i) {
    sum += nodes[i].value;
}
```

如果 `Node` 很胖（包含很多不会被当前循环使用的字段），每次搬运的缓存行中有效数据占比很低，带宽被浪费。

### 3.4 三类常见 Cache Miss

- Cold/Compulsory Miss：第一次访问，必然 miss
- Capacity Miss：工作集超过缓存容量
- Conflict Miss：映射冲突导致的非必要淘汰

定位瓶颈时区分 miss 类型很重要：数据量太大和映射冲突，优化策略完全不同。

## 4. 多核下的一致性：MESI（及其变种）

多核机器里，每个核心可能缓存同一内存地址的副本。缓存一致性协议确保“看到的数据不会彼此矛盾”。

常见协议是 MESI（有些实现是 MOESI 等扩展）。

- M（Modified）：仅本核持有，已修改，和内存不一致
- E（Exclusive）：仅本核持有，未修改，和内存一致
- S（Shared）：多核可共享，和内存一致
- I（Invalid）：无效

### 4.1 写入共享行发生了什么

当核心 A 要写一个 S 状态缓存行：

1. A 需要先获得该行写权限（Read For Ownership，RFO）
2. 其他核心对应行被失效（变 I）
3. A 才能在本地修改，行通常转为 M

如果多个核心反复写同一行，会频繁失效-抢占，形成“乒乓”。这是伪共享的底层机制。

---

## 5. 一致性不等于同步正确性

很多并发 bug 来自一个误解：

- 缓存一致性协议保证“单地址最终一致”
- 但不保证你的程序跨地址读写顺序符合业务语义

还需要语言层同步原语（`std::atomic`、mutex、condition variable 等）与内存序。

### 5.1 C++ 原子内存序（实务视角）

- `memory_order_relaxed`：只保证原子性，不保证先后可见关系
- `memory_order_release` / `memory_order_acquire`：建立跨线程“发布-获取”关系
- `memory_order_seq_cst`：全局单一顺序，最强也通常最慢

结论：

- MESI 解决“硬件缓存副本一致”，
- C++ 内存模型解决“程序观察顺序正确”。

## 6. 伪共享（False Sharing）

### 6.1 定义

不同线程更新不同变量，但这些变量恰好位于同一缓存行，导致互相使对方缓存行失效。

这是“逻辑上不共享，物理上共享缓存行”。

### 6.2 典型症状

- 核数增加但吞吐不升反降
- `atomic` 计数器看似简单却异常慢
- `perf`/PMU 显示高失效、高 cache-to-cache 传输

### 6.3 错误示例

```cpp
#include <atomic>
#include <thread>

struct Counters {
    std::atomic<long long> a{0};
    std::atomic<long long> b{0};
};

int main() {
    Counters c; // a 和 b 很可能落在同一缓存行

    std::thread t1([&] {
        for (int i = 0; i < 50'000'000; ++i) {
            c.a.fetch_add(1, std::memory_order_relaxed);
        }
    });

    std::thread t2([&] {
        for (int i = 0; i < 50'000'000; ++i) {
            c.b.fetch_add(1, std::memory_order_relaxed);
        }
    });

    t1.join();
    t2.join();
}
```

### 6.4 改进示例（隔离缓存行）

```cpp
#include <atomic>
#include <cstddef>
#include <new>
#include <thread>

#if defined(__cpp_lib_hardware_interference_size)
constexpr std::size_t kLine = std::hardware_destructive_interference_size;
#else
constexpr std::size_t kLine = 64;
#endif

struct alignas(kLine) PaddedCounter {
    std::atomic<long long> value{0};
};

struct Counters {
    PaddedCounter a;
    PaddedCounter b;
};

int main() {
    Counters c;

    std::thread t1([&] {
        for (int i = 0; i < 50'000'000; ++i) {
            c.a.value.fetch_add(1, std::memory_order_relaxed);
        }
    });

    std::thread t2([&] {
        for (int i = 0; i < 50'000'000; ++i) {
            c.b.value.fetch_add(1, std::memory_order_relaxed);
        }
    });

    t1.join();
    t2.join();
}
```

注意：

- `alignas` 解决“起始地址对齐”，不自动保证“数组元素之间不共行”。
- 对数组场景，需确保 `sizeof(T)` 也是缓存行大小的倍数，或显式填充。

## 7. C++ 工程里的常见优化手段

### 7.1 写多热点分片（Sharding）

把一个全局热点计数器改成“每线程/每核一个槽位”，周期性汇总：

- 写路径：无争用（或低争用）
- 读路径：集中汇总

适合监控计数、QPS、日志统计。

### 7.2 数据布局重构

- AoS -> SoA：当你按字段批量处理时
- 热冷分离：高频字段单独紧凑存放，低频字段放旁路结构

目的：提高每个缓存行中的“有效载荷比例”。

### 7.3 避免跨线程频繁写同一行

- 尽量让线程“拥有”自己的写区域
- 队列/环形缓冲区中，生产者与消费者的头尾指针可分离对齐

### 7.4 原子操作与锁的选择

- 单个 `atomic` 在高争用下不一定比 mutex 快
- 如果需要复合不变量，mutex 往往更简单可靠

## 8. 如何排查缓存行问题

先测量，再优化。推荐流程：

1. 构造最小复现（microbenchmark）
2. 固定线程数、绑定 CPU（避免调度噪声）
3. 比较“优化前后”的 wall time、吞吐和 PMU 指标
4. 只改一个变量，避免混入其他因素

### 8.1 Linux 常见观测手段

```bash
# 基础指标
perf stat -e cycles,instructions,cache-references,cache-misses ./your_bin

# 若平台支持，可进一步看 cache-to-cache/失效相关事件
perf c2c record -g ./your_bin
perf c2c report
```

### 8.2 解释结果时的注意点

- 关注趋势，不迷信单次绝对值
- 关注 P50/P95，而不只均值
- 对比不同线程数（1/2/4/8/...）的扩展性曲线

### 8.3 macOS 下的常见做法

在 macOS 上通常不使用 Linux `perf`，可用：

- Instruments 的 Time Profiler / Counters（若硬件与权限支持）
- `sample`、`spindump` 做热点与阻塞初步定位

建议先用 wall time + 吞吐曲线确认是否存在扩展性塌陷，再进入更细粒度硬件计数分析。

---

## 9. 常见误区

- 误区 1：只要用了 `atomic` 就一定快
- 误区 2：只要 64B 对齐就不会伪共享
- 误区 3：缓存一致性协议能替代同步原语
- 误区 4：不做测量就凭经验重构布局

---

## 10. 实战检查清单

- 是否存在多线程频繁写入的共享字段？
- 这些字段是否可能落在同一缓存行？
- 是否可以改成每线程本地写 + 合并？
- 是否把热字段和冷字段分离？
- 是否用基准测试验证过优化收益和回归风险？

---

## 11. 总结

- 缓存行是 CPU 性能和并发性能的基础物理单位。
- 伪共享本质是“缓存行所有权争夺”，不是代码层面的显式共享。
- MESI 保证缓存一致性，但程序正确同步仍依赖 C++ 并发原语与内存序。
- 真正有效的优化是：先测量，再定位，再做数据布局和并发结构调整。
